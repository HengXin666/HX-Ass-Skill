## 设计思路

### 与 Bravely You 的核心差异

这首歌采用 **Aegisub karaoke template** 机制, 与 Bravely You 的 Python 预渲染方式不同。Python 脚本只负责**组装 KFX 模板文件**(处理输入 K 帧 → 匹配样式/角色/特效 → 输出 template ASS), 实际的特效渲染由 Aegisub 执行模板完成。

| 对比项 | Bravely You(Python 预渲染) | ときめきポポロン♪(Template 组装) |
|--------|---------------------------|-------------------------------|
| 构建工具 | Python 直接输出 fx | Python 组装 → Aegisub 执行 |
| 模板机制 | 无模板, 代码直接计算每行 | code/template → Apply Template |
| Python 角色 | 计算所有效果参数+渲染 | 只做歌词分配/合并/注音处理 |
| 效果来源 | 代码手动复现参考文件效果 | 直接复用参考文件的 template 行 |
| 可扩展性 | 每首歌需大量自定义 | 换歌词+调整时间即可 |

### 参考文件分析

参考文件 `ときめきポポロン♪ (心动泡芙♪) - チマメ隊.ass` (12.57MB, 22580行) 是完整的原版 KFX:

- **8 组 Style**:
  - `JP_FX_A` — Verse 段(钟表+指针+旋转进场)
  - `JP_FX_AS` — Verse 上部小字(杯子+热气+缩放弹入)
  - `JP_FX_B` — Bridge 段(星星+流星+月亮+提比滑行)
  - `JP_FX_C` — Chorus 段(跳放大进场+提比跳跃+飞翔兔子+Rabbit House 描边)
  - `JP_FX_CS` — Chorus 上部补充(翻转进场+上方提比)
  - `JP_FX_D` — Ending 段(淡入换色+旗帜+人物图)
  - 各 Style 含对应 `-furigana` 变体

- **三角色配色**: chino=`&HEAA380&`(蓝), maya=`&HDAC95E&`(黄绿), megu=`&H758FFA&`(紫)
- **依赖**: Yutils 库(贝塞尔曲线、shape 处理)
- **特殊行**: DP(三人合唱, 人物图), CB(飞翔兔子+星尘), CA(提比跳跃), firstsyl(提比滑行/星星)

### 2. Python 脚本 (`build_kfx.py`, ~640 行)


#### 核心处理

| 处理项 | 说明 |
|--------|------|
| `\kf` → `\k` | LDDC 的 `\kf` 标签转为 Aegisub 识别的 `\k` |
| LINE_MAP | 76 行歌词 → (Style, Actor, inline_fx) 分配 |
| DP_MERGE | 多行合并为 DP 合唱行(per-group `\k` tag) |
| MERGE_LINES | Bridge 段相邻行合并(带 inline_fx 切换) |
| EMBEDDED_PAREN | 括号内容分离为独立 CS 行(独立时间戳) |
| 注音(furigana) | 汉字→假名映射, 使用 `字|<読み` 管道符语法 |
| _fix_consecutive_gaps | 后处理确保连续 JP_FX_B 行时间无间隙 |

### 4. 特殊处理

#### DP (三人合唱) 行
原版 DP 行结构: `{\k70\-megu}可愛い{\k70\-maya} 美味しい{\k301\-chino} おそろいの時間`
- **每个角色组**只有 1 个 `\k` tag (K值 = 组内所有字的 K 值之和)
- 总共 3 个 syl → `math.fmod(syl.i-1,3)+1` 循环 1→2→3 对应图片
- 输入文件中 DP 是分行的(每角色一行), 需要合并

#### 飞翔兔子 (CB)
- 通过 `inline_fx="CB"` 标记触发
- code syl 计算贝塞尔路径: `bezier_n_uniform_speed(x,y,8)` (兔子) 和 `bezier_n_uniform_speed(x,y,1)` (星尘)
- template 用 `fx CB` 过滤

#### 括号词分离 (EMBEDDED_PAREN)
- 某些行包含括号内容(如 `飛び出した`)
- 括号内容分离为 `JP_FX_CS` 样式的独立行
- CS 行的起始时间 = 主行开始 + 括号前所有 `\k` 值之和(确保延迟出现)

## 遇到的问题与解决方案

### 问题 1: DP 人物图顺序错误 + 图片无法加载

**现象**: 三角色人物图 (1.png, 2.png, 3.png) 显示顺序错误, 且某张图片无法加载。

**根因**: `build_dp_line()` 生成了**逐字** `\k` tag (每字一个 syl), 导致 `syl.i` 有大量值, `math.fmod(syl.i-1,3)+1` 的循环与原版完全不同。原版 DP 行只有 3 个 syl (per-group), 所以 `syl.i` 从 1 到 3, 完美对应 3 张图片。

**修复**:
1. 重写 `build_dp_line()`: per-GROUP 模式, 每个角色组只生成 1 个 `\k` tag (K值取组内之和)
2. template 行恢复原版: `fxgroup DP` + `retime("syl",0,0)` + `math.fmod(syl.i-1,3)+1`
3. code syl 简化: `fxgroup.DP=(line.actor=="DP")` (移除复杂的 group tracking 逻辑)

```python
# Before (per-char, 错误):
# {\k23\-megu}可{\k24\-megu}愛{\k23\-megu}い{\k22\-maya}美{\k21\-maya}味...
# → syl.i = 1,2,3,4,5... → math.fmod 循环错乱

# After (per-group, 正确):
# {\k73\-megu}可愛い {\k67\-maya}美味しい {\k160\-chino}おそろいの時間
# → syl.i = 1,2,3 → math.fmod(syl.i-1,3)+1 = 1,2,3 ✓
```

**经验**: 当原版 template 依赖 `syl.i` 做索引循环时, karaoke 源行的 syl 数量必须匹配。**先看 template 怎么用 syl.i, 再决定 karaoke 行的 `\k` 分割粒度。**

### 问题 2: 飞翔兔子特效掉帧

**现象**: 飞翔兔子 + Rabbit House 描边效果卡顿/掉帧, 与原版的流畅效果差异明显。

**根因**: 我们在 code syl 上加了 `fxgroup.CB2=(syl.inline_fx=="CB" and syl.i%2==1)` 条件 + `if fxgroup.CB2 then ... end` 包裹, 只为奇数 syl 计算路径。而原版的 code syl **无任何 fxgroup 条件**, 对所有 syl 都计算贝塞尔路径。template 行用 `fx CB` (基于 `inline_fx`) 过滤, 不需要额外的 fxgroup。

**修复**:
1. code syl: 移除 `fxgroup.CB2` 条件, 直接对所有 syl 计算路径
2. template: `fxgroup CB2` → `fx CB` (恢复原版)

```lua
-- Before (有条件, 掉帧):
-- code syl: fxgroup.CB2=(syl.inline_fx=="CB" and syl.i%2==1); if fxgroup.CB2 then ... end
-- template: template syl noblank notext fxgroup CB2

-- After (无条件, 流畅):
-- code syl: local x,y={},{} ... RR_pos=bezier_n_uniform_speed(x,y,8) ...
-- template: template syl noblank notext fx CB
```

**经验**: **`fx <name>` 和 `fxgroup <name>` 是不同的机制。** `fx` 基于 `inline_fx` (karaoke 源行中 `\-<name>` 标记) 过滤, `fxgroup` 基于 code syl 中设置的 `fxgroup.<name>` 布尔值过滤。当原版用 `fx CB`, 不要改成 `fxgroup CB2` — 这会改变过滤逻辑。code syl 中的计算不需要条件保护, 因为 template 的 `fx` 过滤已经负责了筛选。

### 问题 3: 提比图跨行断开

**现象**: JP_FX_B 段中, 提比(tipi.png)滑行效果在连续行之间有明显断开(消失再出现), 但原版是持续连贯的。

**根因**: 提比模板使用 `retime("line",-100,150)`, 提供 250ms 的时间容错。原版的连续 JP_FX_B 行时间紧密(甚至有 20~60ms 重叠), 但我们输入文件中相邻行之间有 120~280ms 间隙, 超过了 250ms 容错, 导致提比在行间消失。

**修复**: 在 `build_kfx.py` 中添加 `_fix_consecutive_gaps()` 后处理:
- 查找同一 style 的连续行
- 如果间隙 < 5秒(同 section), 自动延长前一行的结束时间, 确保 300ms 重叠
- 间隙 > 5秒(跨 section)不处理, 避免错误连接

```python
def _fix_consecutive_gaps(lines, target_style):
    OVERLAP_MS = 300   # 确保平滑过渡的重叠量
    MAX_GAP_MS = 5000  # 只修复同 section 内的间隙
    # ... 查找 target_style 行, 计算间隙, 延长结束时间
```

**经验**: 当 template 的 `retime` 只提供有限的时间容错时, **输入数据的行间隙必须小于这个容错值**。如果输入数据来自不同的标注源(如 LDDC), 时间戳可能与原版不完全一致, 需要后处理来消除间隙。**关键指标: 计算 retime 提供的总容错 = |pre_offset| + |post_offset|, 确保行间隙 < 该值。**

### 问题 4: 括号词出现时间过早

**现象**: 从主行中分离出的括号内容(如 `飛び出した`, `追い越しちゃって`)与主行几乎同时出现, 但原版中这些内容应该明显晚于主行出现(延迟约 4 秒)。

**根因**: CS 行使用了与主行**完全相同的 start/end 时间戳**。CS 模板的 `retime("abs",$lstart+$sstart-250,$lend)` 中, `$sstart` 是相对于 CS 行自身 start 的偏移。由于 CS 行 start = 主行 start, 第一个 syl 的 `$sstart ≈ 0`, 导致 CS 内容在 `$lstart - 250` 就出现了。

而原版的 CS karaoke 源行有**独立的、偏后的时间戳** — CS 行的 start 比主行晚约 4 秒:
```
# 原版: 主行 0:00:42.60 ~ 0:00:48.24, CS行 0:00:46.52 ~ 0:00:48.24 (偏移 ~3.9s)
# 我们: 主行 0:01:05.16 ~ 0:01:11.04, CS行 0:01:05.16 ~ 0:01:11.04 (无偏移! ❌)
```

**修复**: CS 行的起始时间独立计算:
```
CS_start = 主行start + 括号前所有 \k 值之和(厘秒→毫秒)
CS_end = 主行end (不变)
```

```python
# 遍历 \k tags, 累加括号前的 K 值
pre_paren_k_sum = 0  # centiseconds
for each \k tag before '(':
    pre_paren_k_sum += k_val

cs_start_ms = ts_to_ms(main_start) + pre_paren_k_sum * 10
```

验证结果:
| 主行 start | CS start | 偏移 | 原版偏移 |
|-----------|----------|------|---------|
| 0:01:05.16 | 0:01:09.21 | 4.05s | ~3.92s ✓ |
| 0:02:32.82 | 0:02:36.70 | 3.88s | ~3.89s ✓ |
| 0:03:44.70 | 0:03:48.83 | 4.13s | ~4.12s ✓ |

**经验**: **CS/分离行的时间戳必须独立于主行**, 不能简单复制。正确做法: 从 `\k` 值的累计和来计算分离内容在原行中的绝对开始时间, 然后用这个时间作为分离行的 start。

## 经验教训

### ⭐ 1. Template 组装模式下, syl 分割粒度必须匹配 template 的使用方式

**这是本次最大的坑。**

当 template 行依赖 `syl.i` 做索引计算(如 `math.fmod(syl.i-1,3)+1`)时, karaoke 源行的 `\k` 分割数量**直接决定了 syl.i 的范围**。如果 template 期望 3 个 syl, 就必须生成 3 个 `\k` tag。

**正确流程**:
1. 先看 template 怎么使用 `syl.i` / `syl.inline_fx` / `line.kara.n`
2. 再决定 karaoke 行的 `\k` 分割粒度
3. 不要想当然地按字分割 — 有时按词组/角色分割才是正确的

### ⭐ 2. `fx <name>` vs `fxgroup <name>` 是完全不同的机制

| | `fx <name>` | `fxgroup <name>` |
|---|---|---|
| 数据来源 | karaoke 源行中的 `\-<name>` 标记 | code syl 中设置 `fxgroup.<name>=true/false` |
| 过滤级别 | syl 级 (每个 syl 可以有不同 inline_fx) | syl 级 (code syl 对每个 syl 计算条件) |
| 适用场景 | 根据歌词内容静态标记 | 根据运行时条件动态决定 |

**不要混用**: 当原版用 `fx CB` (基于 inline_fx 标记), 不要改成 `fxgroup CB2` + code syl 条件 — 这会改变过滤语义, 且 code syl 中的计算不需要 fxgroup 条件保护(template 的 `fx` 已经负责筛选)。

### ⭐ 3. 输入数据的行间时间必须匹配 template 的 retime 容错

当 template 使用 `retime("line",-100,150)` 时, 总容错 = 100 + 150 = 250ms。如果相邻行间隙 > 250ms, 效果就会断开。

不同输入源(LDDC vs 手工标注)的时间戳可能不一致, 需要**后处理**来消除间隙:
- 只修复同 section 内的小间隙(< 5s)
- 通过延长前一行的 end 来创建重叠
- 跨 section 的大间隙不要碰

### ⭐ 4. 分离行(CS)的时间戳必须独立计算

从主行中分离出的内容(如括号词), 其 karaoke 源行的时间戳**不能复制主行的时间**。正确做法:
```
CS_start = 主行start + Σ(括号前所有\k值) × 10ms
CS_end = 主行end
```

这样 template 中的 `$sstart` (相对于 CS 行 start 的偏移) 就是 0, 而 `$lstart` (CS 行的绝对 start) 已经是延迟后的时间, 效果自然正确。

### 5. karaskel 的 syl.i 从 1 开始

Aegisub karaskel 中, `syl.i` 的第一个音节是 **1**(不是 0)。所以:
- `syl.i == 1` → 第一个音节 (用于 `fxgroup.firstsyl`)
- `math.fmod(syl.i-1, N)+1` → 从 1 到 N 循环

### 6. 注音(furigana)的 karaskel 限制

每个 `\k` tag 只支持**一个管道符** `|`:
```
{\k30}字|<読み   ← 正确: 一个 \k, 一个 |
{\k30}字1字2|<読み1読み2   ← 错误: karaskel 只取第一个 | 分割
```

因此, 当一个 `\k` group 包含多个汉字时(如 DP 行的 per-group 模式), **不能使用 per-char furigana**, 需要:
- 不加注音, 或
- 使用整词注音(如果需要的话)

### 7. 始终对比原版验证

开发过程中, 每次修改后应:
1. 搜索输出文件中特定字符的行, 与原版对比
2. 检查行数变化(是否与预期一致)
3. 验证时间戳偏移(精确到 10ms 级别)
4. 在 Aegisub 中实际预览效果

### 8. Python 运行时使用 `-X utf8`

Windows 系统默认编码不是 UTF-8, 处理日语/中文文件名和内容时必须加 `-X utf8` 参数:
```bash
python -X utf8 build_kfx.py
```

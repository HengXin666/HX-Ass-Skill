## 设计思路

### 与前三首的核心差异

这首歌**完全不使用 Aegisub karaoke template**。前三首（羽ばたきのバースデイ、INNOCENT BLUE、大切がきこえる）都是"组装 KFX 模板文件 → Aegisub 执行模板 → 生成 fx"的流程, 而 Bravely You 的参考文件是**纯手工逐行制作的 pre-rendered 效果**, 包含手工定位的星星、波纹、回音、配对吟唱等, **无法通过 template 机制复现**。

因此采用 **Python 脚本直接生成所有 fx Dialogue 行**, 输出即最终成品。

| 对比项 | 前三首(模板流程) | Bravely You(Python 预渲染) |
|--------|-----------------|---------------------------|
| 构建工具 | PowerShell + Aegisub | Python 脚本直接输出 |
| 模板机制 | code/template → Apply Template | 无模板, 代码直接计算每行 |
| 效果复杂度 | 参考模板定义的 Lua 效果 | 手工复现参考文件的精确效果 |
| 输出行数 | ~200 行(模板) → ~31000 行(fx) | 直接 5346 行(fx) |
| 可扩展性 | 换歌词即可 | 需要在代码中定义每段的颜色/回音/分段 |
| 中文特效 | 简单 fad(300,300) | clip 切片入场 + 垂直碎裂退场(8行/字) |

### 参考文件分析

参考文件 `Lia - Bravely You.ass` (251KB, 6600+行) 是纯手工制作的 Charlotte OP 效果:

- **31 个 Style**: OPJP(主歌词), OPJP-A/C(快速吟唱左右), OPJP-B(回音), OPCN(中文翻译), OPCN-A/C(中文吟唱左右) 等
- **字体**: OPJP=DFPHSGothic-W5, OPCN=華康宗楷體W7, 吟唱=DFPFreeRyuyou-Lt
- **效果层级**: 每音节 5 层(Main L2, Star L4, Ring×2 L0, Glow L1)
- **颜色**: 4 色系统 — 蓝紫(`&HBA4561&`), 紫(`&H672791&`), 绿(`&H45C043&`), 青(`&HA06E31&`), 按段落/词组分配

## 遇到的问题与解决方案

### 问题 1: 星星和波纹随机位置偏移太小

**现象**: 星星和波纹位置几乎重叠在文字正上方, 缺乏参考文件中的随机自然感。

**根因**: 初始偏移范围太小 (X: ±15px, Y: ±12px), 且 Y 方向没有向下偏移的倾向。

**修复**: 增大偏移范围并添加向下偏移倾向:
```python
# Before:
rx = randint(-15, 15); ry = randint(-20, 5)

# After:
rx = randint(-20, 20); ry = randint(-10, 25)  # Y偏下
```

### 问题 2: 中文字幕消失与下一行出现重叠

**现象**: 上一行中文字幕的退场动画(垂直碎裂)和下一行的入场动画(水平切片)在时间上重叠, 视觉混乱。

**根因**: `enter_start = ms - 300`(入场提前 300ms 开始) + `enter_end = end_ms`(退场要到 end+300ms 才结束), 导致前一行退场和后一行入场有 ~660ms 重叠。

**修复**: 
- 入场不提前: `enter_start = ms`(不减 300ms)
- 退场在行结束前完成: `enter_end = end_ms - 300`(退场动画 300ms, 在 end_ms 时恰好结束)
- 减小 stagger: ±120ms → ±60ms

### 问题 3: 分段中文(word-by-word)的 delay 值错误

**现象**: 中文词的出现时间与参考文件完全不同, 延迟了约 2000ms。

**根因**: **delay 值误用了回音的 echo_delay(~2040ms 起), 但应该使用参考文件 OPCN Comment 行的 start time 偏移(~370ms 起)**。回音有 ~2100ms 的滞后(先唱后回), 但中文字幕应在主唱开始后很快出现。

**发现过程**: 
1. 提取参考文件中所有 `OPCN` 样式的 Comment 行
2. 发现 Comment start time = 23540ms (line_start=23170, **delay=370ms**)
3. 验证 ENTER fx 行 start = 23240ms = 23540-300 (**精确匹配!**)
4. 结论: delay = Comment_start - line_start, **不是** echo_delay

**修复**: 
```python
# Before (使用 echo_delay):
delay: [2040, 5170, 8460, 11590]

# After (使用参考文件 Comment start time):
delay: [370, 3540, 6790, 10000]
```
入场时间: `word_enter_start = word_appear_ms - 300`（动画在 word_appear 时刻完成）

### 问题 4: 日语 OPJP-A/C 吟唱段未实现配对消失

**现象**: 吟唱段中, 左边(A)歌词消失后才出现右边(C)歌词, 但参考文件中 A 和 C 应该同时可见、同时消失。

**根因**: 中文 OPCN-A/C 已实现配对消失(A 延长到 C.end+600ms), 但日语 OPJP-A/C 仍使用各自的原始 end_ms。

**修复**: 对 OPJP-A/C 添加与 OPCN-A/C 相同的配对消失逻辑:
```python
if is_left_jp:
    # A(左) 延长到 C(右) 的 end + 600ms
    jp_pair_end = next_ol['end_ms'] + 600  
else:
    # C(右) 延长到自己的 end + 600ms
    fx_end_jp = end_ms + 600
```

### 问题 5: Verse 2 缺少回音和分段中文

**现象**: `卑怯だったずっと僕は`(115920ms) 和 `きみはいつも向こう見ずだった`(128640ms) 这两行在 verse2 中, 参考文件中有回音效果, 但脚本只为前两行(23170ms, 35890ms)定义了回音。

**修复**: 
- 新增 115920ms 的 `ECHO_DEFINITIONS`: ヒキョウ → ダッタ → ズット → ボクは
- 新增 115920ms 的 `OPCN_WORD_BY_WORD`: 裹足不前 → 的我 → 不过是个 → 懦夫
- 新增 115920 和 128640 的 per-character `COLOR_MAP_BY_LINE`
- 回音 delay 计算: `echo_delay ≈ JP_word_start + 2100ms`
- 中文分段 delay 计算: `delay ≈ JP_word_start + 370ms`

## 经验教训

### ⭐ 1. OPCN delay 和 echo_delay 是完全不同的两个值

**这是本次最大的坑。**

- `echo_delay` ≈ 日语词开始 + **2100ms** → 回音(OPJP-B)在主唱后 ~2.1 秒出现
- OPCN `delay` ≈ 日语词开始 + **370ms** → 中文翻译在主唱后 ~0.37 秒出现

两者相差约 **1730ms**。混淆会导致中文字幕严重延迟。正确做法: **从参考文件的 OPCN Comment 行 start time 直接提取 delay 值**。

### ⭐ 2. 入场动画的时间语义

`gen_cn_char_fx()` 的 `enter_start_ms` 是动画**开始**时间, 300ms 动画后字完全可见。所以:
- 如果希望字在 `T` 时刻完全可见 → `enter_start = T - 300`
- 如果希望字在 `T` 时刻开始动画 → `enter_start = T`

参考文件使用的是 `enter_start = word_appear - 300`(字在 word_appear 时刻完全可见)。

### ⭐ 3. 避免相邻中文行动画重叠

退场动画 300ms + 入场动画 300ms = 最少需要 600ms 间隔。解决方案:
- 入场从 `ms` 开始(不提前)
- 退场在 `end_ms - 300` 开始, `end_ms` 时结束
- 这样只要 `next_ms >= prev_end_ms`, 就不会重叠

### 4. 配对消失(paired disappearing)需要日语和中文同步

OPJP-A/C 和 OPCN-A/C 都需要实现配对消失:
- A(左) 延长到 C(右) 的 `end_ms + 600ms`
- C(右) 延长到自己的 `end_ms + 600ms`
- A 和 C 在视觉上有 ~0.8s 重叠后同时淡出

**容易遗漏**: 只做了中文配对而忘了日语配对。两者逻辑完全相同, 但需要分别实现。

### 5. Python 预渲染 vs Aegisub Template

**适用场景对比:**

| | Python 预渲染 | Aegisub Template |
|---|---|---|
| 适合 | 手工参考文件、复杂自定义逻辑、条件分支多 | 标准效果模板、可复用的效果库 |
| 优势 | 完全控制每一行, 可调试, 可验证 | 简洁, 换歌词即可, Lua 表达力强 |
| 劣势 | 每首歌需要大量自定义配置 | 受 template 机制限制, 调试困难 |
| 输出大小 | 较小(5346 行) | 较大(31000+ 行, 含大量粒子) |

### 6. 始终对比参考文件验证

开发过程中, **每次修改后都应从输出文件中搜索特定字符的 fx 行, 与参考文件对比**, 确认:
- 时间戳是否正确(精确到 10ms)
- ASS 标签顺序和格式是否匹配
- clip 坐标范围是否合理

### 7. 运行 Python 时使用 `-X utf8`

Windows 系统默认编码不是 UTF-8, 处理日语/中文文件时必须加 `-X utf8` 参数, 否则 print() 会因编码错误崩溃。

### 8. 颜色 per-character 映射的局限性

当同一行中同一个字符出现多次但属于不同词组（如 `だった...ずっと` 中 'っ' 出现两次）时, dict 映射无法按位置区分, 只能给同字符同一颜色。可接受的折衷, 除非改为位置映射架构。
